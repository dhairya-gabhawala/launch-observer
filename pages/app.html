Overview

              
Launch Observer helps you see which analytics calls are firing on a website. It is designed for Adobe Experience Platform (AEP) Edge and Adobe Analytics, but you can also use it for other tracking services.

              
Use it when you want to confirm that a page view or click is sending data, and to read the exact values being sent.

              

              
Key concepts (simple definitions)

                

                  
Session: A recording window that keeps related requests together.

                  
Site: A label you assign (for example, “Production” or “Staging”).

                  
Allowlist: The domains you want to capture (for example, AEP Edge).

                  
Requests: The network calls sent by the page.

                  
Payload: The data inside a request, shown in a readable format.

                  
UAT assertions: Rules that validate your analytics data.

                

              

              
            

            

              
Getting Started

              
    
                
  1. Open Launch Observer from your browser toolbar.
    2. 
                
  2. Click Start New Session.
    3. 
                
  3. Choose a Site or create a new one.
    4. 
                
  4. Give the session a friendly name (optional but recommended).
    5. 
                
  5. Pick a tab to listen to (or choose “All tabs”).
    6. 
                
  6. Click Start.
    7. 
                
  7. Now browse the site. Requests will appear as you interact.
    8. 
              

              
            

            

              
Common Tasks

              
    
                
  1. Find a request: Use the search bar at the top of the report to filter by domain, path, or value.
    2. 
                
  2. Read a payload: Click a request in the list. Use the Payload tab to see values in a readable format.
    3. 
                
  3. Check a page view or click: Look for the event type inside the request to confirm a page view or link click.
    4. 
                
  4. Add a custom domain: Open Manage Allowlist, add the domain, and optionally map it to a known service.
    5. 
                
  5. Pause a session: Use the pause icon beside a session to stop listening.
    6. 
                
  6. Create UAT assertions: Download the template from UAT Assertions, open it in a text editor, and add rules for your site (examples: event type exists, page name equals, only one pageview per page).
    7. 
                
  7. Import UAT assertions: In UAT Assertions, select the site, upload the JSON, and click Import.
    8. 
                
  8. Validate UAT assertions: Open UAT Assertions, import your site’s JSON, then enable Perform UAT validations when starting a session.
    9. 
                
  9. Review assertions: Click See Assertions to inspect the active rules and download the JSON if needed.
    10. 
                
  10. Export a UAT report: Use UAT Report to review pass/fail results and export a shareable PDF.
    11. 
              

              
            

            

              
UAT Assertions

              
UAT assertions are JSON rules that validate analytics calls for a site. Each assertion contains one or more conditions, plus optional page-level count rules.

              

                
Top-level structure

                
    
                  
  • siteId and siteName: identify the site.
    • 
                  
  • assertions: array of assertion objects.
    • 
                

              

              

                
Assertion fields

                
    
                  
  • id, title, description
    • 
                  
  • logic: all or any
    • 
                  
  • scope: request or page
    • 
                  
  • conditions: list of checks for payload, query, headers, or raw body
    • 
                  
  • count + value: page-level rules (exactly/at_least/at_most)
    • 
                

              

              

                
Condition fields

                
    
                  
  • source: payload, query, headers, raw
    • 
                  
  • path: dotted path or array index (example: events[0].xdm.eventType)
    • 
                  
  • operator: exists, equals, contains, starts_with, ends_with, regex, in, not_in, gt, gte, lt, lte, range
    • 
                  
  • expected: required for most operators (string, number, or array)
    • 
                

              

              

                
Schema notes (required vs optional)

                
    
                  
  • siteId (required): string identifier for the site.
    • 
                  
  • siteName (optional): human-friendly site label (string).
    • 
                  
  • assertions (required): array of assertion objects.
    • 
                  
  • assertion.id (required): unique string.
    • 
                  
  • assertion.title (optional): string shown in the UI.
    • 
                  
  • assertion.description (optional): string shown in the UI.
    • 
                  
  • assertion.logic (optional): all or any (default: all).
    • 
                  
  • assertion.scope (optional): request or page (default: request).
    • 
                  
  • assertion.conditions (required): array of condition objects.
    • 
                  
  • assertion.count (required only for scope=page): exactly, at_least, or at_most.
    • 
                  
  • assertion.value (required only for scope=page): number.
    • 
                  
  • condition.source (optional): payload, query, headers, or raw (default: payload).
    • 
                  
  • condition.path (required unless source=raw): string path.
    • 
                  
  • condition.operator (required): string operator.
    • 
                  
  • condition.expected (required for comparison operators): string, number, or array.
    • 
                

              

              

                Tip: Use Download template from the UAT Assertions dialog to get a starter file.
              

              

                
{
  "siteId": "example-site",
  "siteName": "Example Site",
  "assertions": [
    {
      "id": "pageview-once",
      "title": "Pageview fires only once",
      "description": "Ensure pageview event fires only once per page view.",
      "logic": "all",
      "conditions": [
        {
          "source": "payload",
          "path": "events[0].xdm.eventType",
          "operator": "equals",
          "expected": "web.webpagedetails.pageViews"
        }
      ],
      "scope": "page",
      "count": "exactly",
      "value": 1
    },
    {
      "id": "page-name-present",
      "title": "Page name is present",
      "logic": "all",
      "conditions": [
        {
          "source": "payload",
          "path": "events[0]._experience.analytics.customDimensions.eVars.eVar41",
          "operator": "exists"
        }
      ]
    }
  ]
}

              

            

            

              
Troubleshooting

              
    
                
  1. I don’t see any requests: Check that the correct domain is in the Allowlist.
    2. 
                
  2. Payload looks empty: Enable “payload capture via page hooks” in the Allowlist settings.
    3. 
                
  3. Only some tabs show data: Make sure your session isn’t locked to a different tab.
    4. 
                
  4. I want a fresh start: Use Clear All Sessions (this does not delete Site names).
    5. 
              

              
            

            

              
FAQ

              

                

                  
Does this change analytics data?

                  
No. Launch Observer is read-only.

                

                

                  
Will it change what is sent?

                  
No. It only reads what the page sends.

                

                

                  
What is a UAT assertion?

                  
A rule that checks whether a request contains the values you expect.

                

                

                  
Do UAT assertions edit or block requests?

                  
No. They only evaluate what the page already sent.

                

                

                  
What happens if a request does not match any assertion?

                  
It shows no UAT badge and is treated as not applicable.

                

                

                  
Can I reuse the same UAT file across sites?

                  
Yes, but it is best to create one JSON per site and update the siteId/siteName.

                

                

                  
Can I validate “only one pageview per page”?

                  
Yes. Use a page-scoped assertion with count set to exactly and value set to 1.

                

                

                  
Can I use it outside Adobe?

                  
Yes. Add any domain to the Allowlist.

                

                

                  
Are values changed?

                  
No. Keys and values keep their original casing.
